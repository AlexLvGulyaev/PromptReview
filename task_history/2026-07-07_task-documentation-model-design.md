# Задача: проектирование целевой модели документации Prompt Review Service

**Дата:** 2026-07-07
**Статус:** В работе

---

## Исходный текст задания

### Контекст

Документационный аудит завершён и утверждён.

В ходе аудита:

- выявлены несоответствия стандартам APL;
- уточнена карта Source of Truth проекта;
- зафиксировано новое правило о границе между инженерной средой APL и публичным GitHub-репозиторием.

### Source of Truth

Использовать:

- действующий CLAUDE.md;
- стандарты из `shared/patterns/`;
- шаблоны из `shared/documentation-standards/`;
- результаты утверждённого аудита документации Prompt Review.

Не выполнять повторный аудит.

Не возвращаться к уже принятым архитектурным решениям без необходимости.

### Цель

Разработать целевую структуру публичной документации проекта Prompt Review Service, соответствующую:

- стандартам APL;
- трёхслойной модели документации;
- принципу Source of Truth;
- правилу разделения инженерной среды APL и публичного GitHub-репозиторием.

### Требования

#### 1. Построить карту публичной документации

Определить полный состав документов, которые должны входить в публичный GitHub-репозиторий.

Для каждого документа указать:

- назначение;
- целевую аудиторию;
- слой документации;
- основной Source of Truth;
- связи с другими документами.

#### 2. Определить жизненный цикл информации

Для каждого типа информации определить:

- где информация рождается (SOT);
- где используется повторно;
- где допускается краткое описание;
- где описание недопустимо во избежание дублирования.

Особое внимание уделить:

- API;
- JSON Contract;
- архитектуре;
- сценариям использования;
- развёртыванию;
- эксплуатации.

#### 3. Проверить публичную самостоятельность документации

Убедиться, что пользователь GitHub сможет:

- понять назначение проекта;
- развернуть систему;
- использовать её;
- сопровождать её;
- развивать её,

не имея доступа к внутренней инженерной среде APL.

Если для этого какого-либо документа не хватает — указать его.

#### 4. Определить судьбу существующих документов

Для каждого текущего документа принять одно из решений:

- оставить без изменений;
- переработать;
- разделить;
- объединить;
- исключить из публичного репозитория.

Каждое решение кратко обосновать.

#### 5. Подготовить план реализации

После утверждения целевой модели предложить порядок выполнения работ.

Сначала должны изменяться документы, являющиеся Source of Truth, затем — производные документы.

Избегать повторной переработки одних и тех же материалов.

### Ограничения

На данном этапе не изменять документацию проекта.

Результатом должна стать архитектура документации и последовательность её реализации.

### Результат

Предоставить:

- целевую структуру публичной документации Prompt Review Service;
- карту связей между документами;
- перечень документов, остающихся во внутренней инженерной среде APL;
- план последовательной переработки документации.

### 2. Жизненный цикл информации

#### 2.1 API и JSON Contract

```
schemas.py (SOT)
    ↓
API_CONTRACT.md (первичная документация)
    ↓
API_DOCS.md (расширенная документация с примерами)
    ↓
README.md (краткое упоминание со ссылкой)
```

**Правила:**
- `schemas.py` — единственный источник истины для полей, типов, валидации
- `API_CONTRACT.md` — формальное описание допустимых значений (quality_level, review_mode)
- `API_DOCS.md` — пользовательская документация с примерами запросов/ответов
- `README.md` — только ссылка на API_DOCS.md, без дублирования примеров

**Допустимые значения quality_level:**
```python
# schemas.py (SOT)
class QualityLevel(str, Enum):
    EXCELLENT = "excellent"
    GOOD = "good"
    FAIR = "fair"
    POOR = "poor"
    NOT_APPLICABLE = "not_applicable"
```

**Допустимые значения review_mode:**
```python
# schemas.py (SOT)
class ReviewMode(str, Enum):
    STANDARD = "standard"
    DETAILED = "detailed"
```

#### 2.2 Архитектура

```
main.py, adapters/*.py (SOT)
    ↓
ARCHITECTURE.md (описание архитектуры)
    ↓
integration-diagram.md (визуализация)
    ↓
README.md (краткое упоминание со ссылкой)
```

**Правила:**
- Код — первичный источник архитектурных решений
- `ARCHITECTURE.md` — описание без дублирования кода
- `integration-diagram.md` — визуальное представление архитектуры
- `README.md` — только высокоуровневая схема со ссылкой

#### 2.3 Сценарии использования

```
Реальная система (SOT)
    ↓
SYSTEM_DEMO.md (скриншоты работы)
    ↓
USER_GUIDE.md (инструкция пользователя)
    ↓
README.md (краткое описание со ссылкой)
```

**Правила:**
- Скриншоты делаются с реальной системы
- `SYSTEM_DEMO.md` — визуальная демонстрация
- `USER_GUIDE.md` — пошаговая инструкция
- `README.md` — только ссылка на USER_GUIDE.md

#### 2.4 Развёртывание

```
docker-compose.yml, .env.example (SOT)
    ↓
DEPLOYMENT_GUIDE.md (инструкция)
    ↓
OPERATOR_GUIDE.md (эксплуатация)
    ↓
README.md (краткое упоминание со ссылкой)
```

**Правила:**
- Docker-compose и .env.example — первичные источники конфигурации
- `DEPLOYMENT_GUIDE.md` — подробная инструкция развёртывания
- `OPERATOR_GUIDE.md` — эксплуатационное руководство
- `README.md` — только ссылка на DEPLOYMENT_GUIDE.md

#### 2.5 Эксплуатация

```
docker-compose.yml, логи, мониторинг (SOT)
    ↓
OPERATOR_GUIDE.md (руководство оператора)
    ↓
CHANGELOG.md (история изменений)
    ↓
README.md (не содержит эксплуатационных деталей)
```

**SOT → первичная документация → производная документация**

1. Сначала изменяем документы, которые являются источниками истины
2. Затем документы, которые извлекают информацию из SOT
3. Затем документы, которые ссылаются на первичную документацию
4. В конце — README.md (самый производный документ)
