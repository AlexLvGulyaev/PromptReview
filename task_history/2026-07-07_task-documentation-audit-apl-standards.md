# Задача: аудит и приведение документации Prompt Review Service к стандартам APL

**Дата:** 2026-07-07
**Статус:** Аудит завершён, ожидает решения

---

## Исходный текст задания

### Контекст

Проект Prompt Review Service завершил этап реализации MVP, опубликован на GitHub и успешно прошёл функциональную проверку.

По итогам уроков PEl03–PEl06 накоплены:

- замечания Антона
- инженерные улучшения проекта
- стандарты документации AI Automation Portfolio Lab
- шаблоны эксплуатационной документации

Кроме того, в ходе развития APL была сформирована единая система документации лаборатории.

Source of Truth по документационным стандартам расположен на уровне APL, а не внутри проекта Prompt Review.

Перед началом дальнейшего развития проекта необходимо проверить документацию Prompt Review на соответствие этим стандартам.

### Source of Truth

При выполнении задачи использовать исключительно действующие стандарты APL.

В первую очередь:

- CLAUDE.md
- shared/patterns/
- shared/documentation-standards/

Не создавать внутри Prompt Review собственные версии общих стандартов.

Если обнаружено дублирование стандартов APL внутри проекта — зафиксировать его как замечание.

Также использовать:

- вердикты Антона по PEl03–PEl05
- текущую документацию Prompt Review
- фактическую реализацию проекта

Структура APL подтверждает, что стандарты документации являются общими активами лаборатории и применяются всеми проектами.

---

## Этап 1. Документационный аудит

Провести полный аудит документации Prompt Review.

Определить:

- какие документы существуют
- какую аудиторию обслуживает каждый документ
- какие документы являются Source of Truth
- какие документы являются производными
- какие документы отсутствуют
- какие документы дублируют друг друга

Особое внимание уделить README.

Проверить, не содержит ли README инженерной информации, которая должна находиться в других документах.

## Этап 2. Проверка соответствия стандартам APL

Проверить соответствие документации:

- трёхслойной модели документации
- обязательному пакету эксплуатационных документов
- правилам размещения артефактов
- принципам Source of Truth

По каждому обнаруженному несоответствию указать:

- что нарушено
- какой стандарт нарушен
- рекомендуемый способ устранения

Не выполнять исправления автоматически.

Сначала предоставить результаты аудита.

## Этап 3. Проверка замечаний Антона

Сопоставить текущую реализацию проекта с замечаниями по PEl03–PEl05.

Отдельно проверить:

- единый JSON-контракт
- документирование допустимых значений quality_level
- документирование допустимых значений review_mode
- статус перехода к общей JSON Schema
- состояние ReAct-подхода
- соответствие текущей реализации ранее полученным рекомендациям

Для каждого замечания определить статус:

- реализовано
- реализовано частично
- не реализовано
- потеряло актуальность

## Этап 4. Проверка Source of Truth проекта

Для Prompt Review определить перечень документов Source of Truth.

Например:

- архитектура
- API
- JSON Contract
- OpenAPI
- PROJECT_STATE
- SPEC

Для остальных документов определить, какие именно SOT они используют.

Никакой документ не должен иметь собственного источника истины, противоречащего SOT проекта или APL.

## Этап 5. Предложение новой структуры документации

После завершения аудита предложить обновлённую структуру документации Prompt Review.

Для каждого документа указать:

- назначение
- целевую аудиторию
- уровень документации
- источник истины
- необходимость переработки

При этом:

- не переносить стандарты APL внутрь Prompt Review
- не создавать локальные копии общих документов лаборатории
- использовать ссылки на общие стандарты там, где это возможно

---

## Ограничения

На данном этапе не изменять документацию проекта.

Задача текущего этапа — исключительно аудит, выявление несоответствий и подготовка плана реорганизации.

Любые изменения структуры документации должны выполняться только после утверждения результатов аудита.

---

## Результат

Предоставить:

- отчёт по аудиту документации
- перечень выявленных несоответствий
- карту Source of Truth для Prompt Review
- предложение по новой структуре документации
- план поэтапного приведения проекта к стандартам APL

### 2.4 Принципы Source of Truth (SOT)

**Стандарт APL (documentation-source-of-truth-discipline.md):**

1. **Иерархия SOT:**
   - БД (schema, данные) → Workflow файлы → Код → JSON Schema

2. **Запрещено:**
   - Выдумывать промпты, модели, параметры
   - Создавать синтетические примеры данных
   - Использовать память модели как источник
   - Копировать из других проектов

**Анализ Prompt Review:**

| Проверка | Статус | Комментарий |
|----------|--------|-------------|
| Промпты извлечены из SOT | ❓ Требуется проверка | Нужно сверить с кодом |
| Модели указаны корректно | ✅ Корректно | Указаны OpenAI, Ollama |
| JSON-контракт соответствует коду | ❓ Требуется проверка | Нужно сверить с schemas.py |
| Примеры данных реалистичны | ✅ Корректно | Используются осмысленные примеры |
| Инфраструктурные данные верны | ❓ Требуется проверка | Нужно сверить с docker-compose |

## ЭТАП 3. ПРОВЕРКА ЗАМЕЧАНИЙ АНТОНА (PEl03–PEl05)

### 3.1 PEl03 — LangFlow MVP

**Вердикт Антона:**
> Предложил сразу заложить более жёсткую структуризацию вывода под интеграции (JSON / Structured Output в самом Flow), чтобы следующий шаг с API и n8n был почти автоматическим. Ещё можно вынести часть логики из системного промпта в отдельные блоки (оценка, рекомендации, переформулировка) — так будет проще поддерживать и доучивать отдельных "подагентов".

**Статус реализации:**

| Рекомендация | Статус | Комментарий |
|--------------|--------|-------------|
| JSON / Structured Output | ✅ **Реализовано** | PEl05: Structured Output в LangFlow, JSON-контракт |
| Вынести логику в блоки | ✅ **Реализовано** | PEl05: модульный конвейер в LangChain Code (extractInput → classifyPrompt → reviewPrompt → rewritePrompt) |

### 3.2 PEl04 — LangChain

**Вердикт Антона:**
> Единственный момент, который можно доработать — я бы в промпте агента убрал жёсткое "Перед анализом используй инструмент" и оставил пространство для решения модели, сейчас это чуть ломает естественный ReAct-паттерн. Ещё можно вернуть из prompt_metrics не строку JSON, а dict, чтобы агенту было проще опираться на отдельные поля.

**Статус реализации:**

| Рекомендация | Статус | Комментарий |
|--------------|--------|-------------|
| Убрать жёсткое требование инструмента | ❓ **Требуется проверка** | Нужно проверить текущий промпт |
| Возвращать dict из prompt_metrics | ❓ **Требуется проверка** | Нужно проверить код |

### 3.3 PEl05 — n8n Integration

**Вердикт Антона:**
> Вынести формат JSON-контракта в один общий артефакт/тип (схему) и в README жёстко зафиксировать варианты значений quality_level и режимов review_mode, чтобы упростить дальнейшее развитие и интеграции. Ещё можно чуть упростить промпты, убрав дублирующиеся пояснения и перенести часть "правил" в комментарии к схеме, чтобы снизить токены и сделать поведение модели более стабильным.

**Статус реализации:**

| Рекомендация | Статус | Комментарий |
|--------------|--------|-------------|

| Зафиксировать quality_level | ❓ **Требуется проверка** | Нужно проверить SPEC и schemas.py |
| Зафиксировать review_mode | ❓ **Требуется проверка** | Нужно проверить SPEC и schemas.py |

## Дополнительные требования (получены в ходе аудита)

### Граница между инженерной средой APL и публичным репозиторием

**Принцип:** Публичный GitHub-репозиторий описывает продукт, а не процесс его разработки.

**Правило добавлено в CLAUDE.md** в ходе аудита.

**Ключевые положения:**

1. **Допускается в публичной документации:**
   - упоминать AI Automation Portfolio Lab как среду разработки проекта;
   - указывать, что проект создавался с использованием инженерных практик и стандартов APL;
   - ссылаться исключительно на документы, входящие в состав публичного репозитория.

2. **Не допускается в публичной документации:**
   - ссылки на `task-history/`
   - ссылки на `PROJECT_GOAL.md`
   - материалы уроков
   - PDF-отчёты по урокам
   - вердикты преподавателей и кураторов
   - материалы Discovery
   - внутренние исследования
   - содержимое `attachments/`

3. **Правило самостоятельности проекта:**
   Любой пользователь, открывший публичный репозиторий, должен иметь возможность понять назначение, развернуть, использовать, сопровождать и развивать проект без доступа к внутренней инженерной среде APL.

**Влияние на аудит:**
Это правило требует дополнительной проверки:
- README.md не должен содержать ссылок на task-history, PROJECT_GOAL, материалы уроков
- Публичная документация должна быть полностью самодостаточной
- Все внутренние артефакты должны быть исключены из GitHub или не упоминаться
