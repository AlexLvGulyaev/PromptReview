### Анализ текущего README.md

**Проблемы:**

#### 1. Нарушение правила публичности

README содержит упоминания внутренних артефактов APL:

```markdown
## История задач
Каталог: task_history/

## Входные материалы
Каталог: attachments/input/
- Module 7 (LangChain & Agents)
- PDF-отчёты по завершённым урокам
```

**Нарушение:** task_history, attachments/input, attachments/reports — внутренние артефакты, не должны упоминаться в публичном README.

#### 2. Инженерные детали вместо продуктового описания

README содержит технические детали, которые должны быть в ARCHITECTURE.md:

```markdown
### Итоговая архитектура
Telegram / Web / n8n / CLI
          ↓
      FastAPI (/review)
          ↓
  Prompt Review Engine
          ↓
LangFlow или LangChain backend
```

**Нарушение:** Архитектурные диаграммы должны быть в ARCHITECTURE.md, а не в README.

#### 3. Детальное описание учебных этапов

README описывает каждый этап PEl03-PEl06 как учебные уроки:

```markdown
### PEl03: LangFlow MVP
**Каталог:** langflow/
**Назначение:** Прототип Prompt Review Agent...
```

**Проблема:** Это выглядит как пересказ учебных уроков, а не история развития продукта.

#### 4. JSON-контракт в README

```markdown
## JSON-контракт
Единый формат ответа для всех реализаций:
```json
{...}
```

**Нарушение:** JSON-контракт должен быть в API_CONTRACT.md, а не в README.

#### 5. Отсутствует продуктовый фокус

README отвечает на вопросы "как реализовано", а не "зачем это нужно".

#### 2. Сценарий повествования

**История развития продукта:**

1. **Проблема** → Качество промптов влияет на результат AI-систем
2. **Решение** → AI-агент анализирует промпты как инженерный объект
3. **Эволюция** → От прототипа (LangFlow) к production-сервису (FastAPI)
4. **Результат** → Публичный API, который можно интегрировать

**Не пересказывать учебные уроки**, а показать путь развития продукта.

#### 3. Уровень детализации

| Уровень | Документ | Содержание |
|---------|----------|------------|
| **Публичный** | README.md | Что это, зачем, как попробовать |
| **Пользовательский** | USER_GUIDE.md (будет создан) | Как использовать |
| **Продуктовый** | SPEC.md | Продуктовая спецификация |
| **Технический** | ARCHITECTURE.md | Архитектура и технические детали |
| **Контрактный** | API_CONTRACT.md | JSON-контракт |

#### 4. Что убрать из README

**Убрать:**
- ❌ Раздел "История задач"
- ❌ Раздел "Входные материалы"
- ❌ JSON-контракт (перенести ссылку в API_CONTRACT.md)
- ❌ Детальное описание PEl03-PEl06 (кратко в "Эволюция проекта")
- ❌ Псевдографику архитектуры (ссылка на ARCHITECTURE.md)
- ❌ Упоминания Module 7, уроков, PDF-отчётов

**Добавить:**
- ✅ Продуктовое описание (какую проблему решает)
- ✅ Скриншоты демо (Web UI, Telegram Bot)
- ✅ Примеры использования
