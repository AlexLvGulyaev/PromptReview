**Предложенная структура:**

### 5. Принцип использования изображений

#### Анализ текущего подхода

**Текущий принцип:**
> «Какие скриншоты есть, те и используем»

**Проблема:**
- Скриншоты вставляются, потому что они существуют
- Нет связи между тезисом документа и изображением
- Изображения не помогают пониманию, а просто присутствуют

#### Предложенный принцип

**Новый принцип:**
> «Какую мысль документа необходимо визуально объяснить?»

**Алгоритм выбора:**
1. Определить тезис документа/раздела
2. Определить, что нужно визуализировать
3. Подобрать изображение, которое **максимально помогает понять тезис**
4. Если подходящего изображения нет — **не вставлять**

**Правило:**
> Скриншоты не вставляются только потому, что они существуют.

#### Решение

**Дополнить MEDIA_INDEX.md правилом использования изображений.**

**Добавить раздел "Принцип использования":**

```markdown
## Принцип использования изображений

### Основной принцип

При выборе изображения исходить из вопроса:

**«Какую мысль данного документа необходимо визуально объяснить?»**

Не из вопроса: «Где можно использовать этот скриншот?»

### Алгоритм выбора

1. Определить тезис документа или раздела
2. Определить, что требует визуализации
3. Подобрать изображение, максимально помогающее понять тезис
4. Если подходящего изображения нет — не вставлять

### Правила

- Скриншоты не вставляются только потому, что они существуют
- Каждое изображение должно объяснять конкретную мысль
- Если изображение не помогает пониманию — удалить

### Матрица использования

| Документ | Тезис | Изображения | Обоснование |
|----------|-------|-------------|-------------|
| README.md | Продемонстрировать продукт | IMG-032, IMG-029 | Web UI и Telegram — главные точки входа |
| USER_GUIDE.md | Научить пользоваться | IMG-030, IMG-031-033, IMG-027-029 | Пошаговые инструкции |
| ARCHITECTURE.md | Объяснить архитектуру | IMG-003, IMG-010-013, IMG-014-021 | Архитектурные диаграммы |
```

## АРГУМЕНТАЦИЯ ПО КАЖДОМУ ИЗМЕНЕНИЮ

### Почему MEDIA_INDEX.md, а не SCREENSHOT_INDEX.md?

**Аргумент 1: Точность**
- JSON не является скриншотом
- Название не соответствует содержимому

**Аргумент 2: Масштабируемость**
- В будущем могут появиться GIF, видео, SVG
- Не потребует переименования

**Аргумент 3: Профессиональность**
- "Media Index" — стандартное название для каталога медиаматериалов
- Соответствует практике других проектов

---

### Почему "result", а не "demo"?

**Аргумент 1: Семантика**
- "Demo" — демонстрация процесса
- "Result" — результат работы
- Изображения показывают результаты, а не процессы

**Аргумент 2: Понятность**
- Через год `result_web_analysis.png` понятнее, чем `demo_web_analysis1.png`
- "Result" указывает на содержимое, "demo" — универсально

**Аргумент 3: Зарезервированность**
- "demo" можно зарезервировать для GIF-анимаций и видео
- Чёткое разделение: result — статичные результаты, demo — динамичные демонстрации

---

### Почему docs/examples/, а не docs/screenshots/?

**Аргумент 1: Логика**
- JSON — это примеры API-запросов/ответов
- Скриншоты — это изображения
- Разные типы контента — разные директории

**Аргумент 2: Расширяемость**
- Можно добавить curl-примеры
- Можно добавить Python-примеры
- Можно добавить JavaScript-примеры

**Аргумент 3: Чистота**
- `docs/screenshots/` — только изображения
- `docs/examples/` — только примеры

---

### Почему не создавать изображения архитектуры?

**Аргумент 1: Mermaid достаточен**
- Архитектурные диаграммы в Mermaid редактируемые
- Версионируются вместе с Markdown
- Не требуют отдельных файлов

**Аргумент 2: Поддерживаемость**
- Изменения в архитектуре требуют перерисовки PNG
- Mermaid обновляется автоматически
