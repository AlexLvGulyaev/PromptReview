# Задача: Архитектурное решение интеграции LangFlow в APL

**Дата:** 2026-07-04
**Статус:** Выполнено

---

## Исходное задание

Принять архитектурные решения по интеграции LangFlow в инфраструктуру APL.

**Цель:** Создать полную и непротиворечивую архитектурную спецификацию, достаточную для начала развёртывания без принятия дополнительных концептуальных решений.

**Ограничения:**
- Не изменять существующие документы
- Не создавать compose
- Не писать код
- Не выполнять deployment
- Документ содержит только архитектурные решения

---

## Принятые архитектурные решения

### 1. Способ развёртывания

**Решение:** Адаптированная конфигурация Docker Compose (2 сервиса).

**Альтернативы:**
- Официальный production compose (11 сервисов) — отклонён из-за избыточности

**Обоснование:**
- Существующий Traefik уже работает
- Prompt Review Flow не требует Celery/RabbitMQ
- Минимальные ресурсы для одного Flow

### 2. База данных

**Решение:** Отдельный PostgreSQL-контейнер с отдельной базой `langflow_db`.

**Альтернативы:**
- SQLite — отклонён (не production-ready)
- Существующий PostgreSQL — отклонён (риск конфликтов схем)

**Обоснование:**
- Изоляция данных LangFlow
- Независимое масштабирование
- Отдельные бэкапы
- Соответствие официальной рекомендации

### 3. Docker-сети

**Решение:** Создать `langflow-network` + подключиться к `n8n_default`.

**Обоснование:**
- `langflow-network` — внутренняя сеть для LangFlow ↔ PostgreSQL
- `n8n_default` — публикация через существующий Traefik
- PostgreSQL не доступен из `n8n_default` (безопасность)

### 4. Поддомен

**Решение:** `langflow.alex-n8n.site`

**Обоснование:**
- Соответствует шаблону `*.alex-n8n.site`
- Уникальный среди существующих (`n8n-de`, `lead-qual`, etc.)
- DNS уже настроен (wildcard)

### 5. Структура файлов

**Создаваемые файлы:**
- `docker-compose.langflow.yml` — конфигурация сервисов
- `.env.example` — шаблон переменных (без секретов)
- `.gitignore` — исключение `.env`

**Неизменяемые файлы:**
- `LANGFLOW_VPS_DEPLOYMENT_PLAN.md` — результаты аудита
- `LANGFLOW_DEPLOYMENT_RESEARCH.md` — исследование документации

### 6. Хранение секретов

**Решение:** `.env` файл, исключённый из Git.

**Обоснование:**
- Стандартный подход для Docker Compose
- `.env.example` коммитится как шаблон
- `.env` в `.gitignore`

### 7. Интеграция с Ollama

**Решение:** `host.docker.internal:11434` через `extra_hosts`.

**Обоснование:**
- Ollama работает на хосте VPS
- Официальная рекомендация LangFlow для Docker
- Флаг `host.docker.internal:host-gateway` в compose

### 8. Интеграция с n8n

**Принцип:** n8n вызывает LangFlow через HTTP API.

**Архитектура:**
- Endpoint: `/api/v1/run/{flow_id}`
- Аутентификация: API Key в заголовке `x-api-key`
- Формат: JSON request/response

### 9. Последовательность внедрения

**11 шагов:**
1. Создание файлов
2. Создание `.env`
3. Создание Docker-сети
4. Запуск PostgreSQL
5. Запуск LangFlow
6. Проверка Health Check
7. Добавление маршрута в Traefik
8. Проверка UI
9. Создание Prompt Review Flow
10. Публикация API
11. Интеграция с n8n

### 10. Критерии готовности

**5 категорий:**
- Контейнеризация (4 критерия)
- Сеть (3 критерия)
- Публикация (3 критерия)
- Функциональность (6 критериев)
- Интеграция с n8n (3 критерия)

## Рассмотренные альтернативы

| Решение | Альтернатива | Причина отклонения |
|---------|-------------|-------------------|
| Адаптированный compose | Официальный production compose | 11 сервисов избыточно для одного Flow |
| Отдельный PostgreSQL | SQLite | Не подходит для production |
| Отдельный PostgreSQL | Существующий PostgreSQL | Риск конфликтов схем |
| `langflow-network` + `n8n_default` | Только `n8n_default` | PostgreSQL должен быть изолирован |
| `host.docker.internal` | IP-адрес хоста | Динамический IP может меняться |
