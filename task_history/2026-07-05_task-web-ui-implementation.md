# Задача: Реализация Web UI для Prompt Review Service

## Исходное задание

Реализовать этап 7 IMPLEMENTATION_PLAN — Web UI для Prompt Review Service.

### Требования:

1. **Архитектура**: Web UI как самостоятельный интерфейс над существующим FastAPI API.
   - Web UI не должен содержать бизнес-логику анализа
   - Вся логика остаётся в Prompt Review Service
   - Web UI вызывает только endpoint `/review`

2. **Использовать существующий endpoint `/review`**:
   - Запрещается обращаться напрямую к LangFlow или LangChain
   - Все запросы через FastAPI API

3. **Пользовательский сценарий**:
   - Краткое описание сервиса
   - Большое поле ввода текста
   - Кнопка «Проанализировать»
   - Индикатор выполнения анализа
   - Отображение результата

4. **Отображение для промпта**:
   - Итоговая оценка качества
   - Оценки по всем критериям
   - Назначение
   - Сильные стороны
   - Слабые стороны
   - Рекомендации
   - Улучшенная редакция
   - Время обработки

5. **Отображение для не-промпта**:
   - Сообщение о том, что текст не распознан как промпт
   - Причина
   - Варианты преобразования в промпт
   - Основные метрики текста

6. **Интерфейс для конечного пользователя**:
   - Не отображать JSON
   - Удобочитаемая структура:
     - Карточки
     - Цветовые акценты
     - Прогресс-бары или визуальные шкалы оценок
     - Отдельный блок улучшенной редакции
     - Адаптивная вёрстка

7. **Обработка ошибок**:
   - Недоступность FastAPI
   - Timeout
   - Сетевые ошибки
   - Ошибки валидации
   - Некорректный ответ
   - Пользователь всегда получает понятное сообщение

8. **E2E-тестирование**:
   - Загрузка страницы
   - Анализ корректного промпта
   - Анализ обычного текста
   - Отображение ошибок API
   - Работа с обоими backend (LangChain и LangFlow)
   - Скриншоты всех сценариев

9. **Документация**:
   - `web/README.md`
   - Инструкции по запуску
   - Описание структуры Web UI
   - Необходимые переменные окружения

10. **Обновление артефактов**:
    - `PROJECT_STATE.md`
    - `README.md`
    - `task-history`

### Дополнительное пожелание:

При проектировании Web UI ориентироваться не только на учебную демонстрацию, но и на портфельный проект. Интерфейс должен выглядеть как самостоятельный продукт, который можно показать потенциальному заказчику или разместить в GitHub как демонстрационное приложение.

### Этап 2: Реализация Web UI

**Созданные файлы:**

```
api/web/
├── index.html      # Главная страница (HTML5, семантическая разметка)
├── styles.css      # Стили (CSS Variables, 15 KB)
├── app.js          # Логика (Vanilla JS, 12 KB)
└── README.md       # Документация
```

**Ключевые решения:**

1. **Дизайн-система на основе Lead Qualification**:
   - Тёмная тема (`--bg-primary: #0a0a0f`)
   - Teal/Cyan акцент (`--accent-primary: #14b8a6`)
   - Цвета для качества: excellent (green), good (teal), fair (orange), poor (red)
   - Шрифты: Outfit (display), Inter (body)

2. **Компоненты**:
   - Карточки с цветными акцентами для статусов
   - Прогресс-бары для визуализации оценок (0–10)
   - Бейджи для качества промпта
   - Теги для приоритетов рекомендаций (high/medium/low)

3. **Адаптивная вёрстка**:
   - Desktop: 3 колонки для оценок
   - Tablet: 2 колонки
   - Mobile: 1 колонка

4. **Обработка ошибок**:
   - Проверка статуса API (health check)
   - Валидация длины текста (10–10000 символов)
   - Обработка network errors, timeout, invalid JSON
   - Понятные сообщения пользователю

5. **Функции**:
   - Автоматический счётчик символов
   - Ctrl/Cmd+Enter для быстрого анализа
   - Копирование improved prompt в один клик
   - Автоматическая генерация user_id (localStorage)
