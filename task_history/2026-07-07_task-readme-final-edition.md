## АРХИТЕКТУРНОЕ РЕШЕНИЕ

### Принцип: Публичный репозиторий не зависит от операционного контура автора

**Принятое решение:**

Публичный GitHub-репозиторий не должен зависеть от существования публичной демонстрационной инфраструктуры автора.

**Причины:**

1. **Операционный контур автора** — публичные сервисы (Web UI, API, Telegram Bot) используют личные API-ключи и вычислительные ресурсы автора, а не являются частью репозитория.

2. **Изменчивость инфраструктуры** — доступность демо со временем неизбежно меняется: сервер, домен, сертификаты, баланс, API-ключи и т.д.

3. **Самодостаточность README** — README должен оставаться полностью актуальным независимо от состояния внешней инфраструктуры.

**Соответствие принципам APL:**

> Публичная документация должна быть самодостаточной и не зависеть от внутренней операционной инфраструктуры проекта.

**Что изменено в README:**

| Было | Стало |
|------|-------|
| Ссылки на публичный Web UI | Скриншоты + инструкция локального запуска |
| Ссылка на публичный API | Пример с localhost |
| Ссылка на Telegram Bot | Описание + скриншот |
| Раздел "Онлайн" | Удалён, акцент на локальный запуск |

**Что осталось:**

- Качественное продуктовое описание
- Архитектурное описание
- Скриншоты
- Инструкция локального запуска
- Ссылки на инженерную документацию

**Главная идея:**

> GitHub демонстрирует инженерное решение, а не предоставляет постоянно работающий SaaS-сервис.
>
> Пользователь должен иметь возможность самостоятельно развернуть проект и получить тот же результат.

## ВОПРОС К ЗАКАЗЧИКУ

**Контекст:** Публичные демо-URL (Web UI, API, Telegram Bot) активны, но используют личные API-ключи автора. При публикации GitHub-репозитория это может привести к расходу лимитов.

**Требуется решение:** Как поступить с публичными URL в README?

**Варианты:**

| Вариант | Описание | Плюсы | Минусы |
|---------|----------|-------|--------|
| **А. Оставить как есть** | URL остаются активными | Максимальная демонстрационная ценность | Риск расхода личных лимитов |
| **Б. Убрать URL** | Полностью удалить публичные URL из README | Безопасность лимитов | Потеря интерактивности, пользователи не могут попробовать |
| **В. Только скриншоты** | Убрать URL, оставить скриншоты | Безопасность, демо-ценность через изображения | Пользователь не может попробовать сам |
| **Г. Docker Compose** | Акцент на локальный запуск | Полный контроль у пользователя | Требует настройки и API-ключа у пользователя |
| **Д. Rate limiting** | Добавить ограничение запросов на API | Баланс между демо и безопасностью | Требует доработки backend |
| **Е. Demo-аккаунт** | Создать отдельный demo-аккаунт OpenAI | Изоляция от личного | Требует настройки и поддержки |

**Рекомендация из анализа:**

Комбинированный вариант (В + Г):
1. В README оставить скриншоты — сохраняется демонстрационная ценность
2. URL закомментировать или убрать — не расходуются лимиты
3. Добавить инструкцию для локального запуска — пользователь может развернуть сам

**Альтернатива для публичного демо:**
- Отдельный demo-аккаунт OpenAI с лимитами free tier
- Rate limiting на уровне API (например, 10 запросов/час на IP)
